How It Works

Four practical steps from incident to production-ready fix.

📋

1. Capture the incident

Paste a real production incident into the form above. Focus on symptoms, impact, and when it started.

🔍

2. Run the 4-agent pipeline

Click “Diagnose”. The Router, Retriever, Skeptic, and Verifier agents classify the failure, simulate evidence, and challenge each other before committing to a result.

✅

3. Review the diagnosis

You get a single, evidence-backed root cause with confidence, fix steps, rollback guidance, and tests to run.

🚀

4. Apply, test, and share

Apply the fix in your environment, run the suggested tests, then export the diagnosis into your runbooks or incident tickets.

How to get the best results

  • Redact secrets, keys, and customer data before you paste any incident.
  • Include a clear timeline: when the problem started, what changed, and which systems are affected.
  • Add 1–3 representative log lines or error messages instead of entire log files.
  • Mention environment and scope (prod vs staging, region, service name) so recommendations stay grounded.
  • Use the export buttons in the results panel to attach the diagnosis to runbooks, postmortems, or incident tickets.

Integrate With Your Stack

Use the web UI for quick diagnosis, or connect programmatically via REST API, webhooks, and batch processing.

Single Incident Diagnosis

Send any production incident and get back a structured root-cause analysis with confidence score, fix plan, rollback steps, and suggested tests.

# Diagnose a single incident
curl -X POST http://localhost:3000/api/diagnose \
  -H "Content-Type: application/json" \
  -d '{"incident": "API latency spiked to 12s in us-east-1 after deploy v2.4.1, connection pool at 98% capacity"}'

# Response includes traceId, 4-agent pipeline results, confidence, fix plan, and rollback

Every response includes an X-Trace-Id header for end-to-end request tracing across your systems.

Batch Diagnosis (up to 100 incidents)

Process up to 100 incidents in a single request. Each incident runs through the full 4-agent pipeline independently. Partial failures are reported per-item, not as a batch failure.

# Batch-diagnose multiple incidents at once
curl -X POST http://localhost:3000/api/batch-diagnose \
  -H "Content-Type: application/json" \
  -d '{
    "incidents": [
      "Memory leak in worker pods after cron job at 02:00 UTC",
      "Database replica lag exceeding 30s during peak traffic",
      "SSL cert rotation failed on edge nodes in eu-west-2"
    ]
  }'

# Returns { batchId, traceId, results: [...], errors: [...] }

Webhook Notifications

Register a webhook URL and receive diagnosis results automatically whenever an incident is processed. Connect to Slack, PagerDuty, or your internal tools.

# Register a webhook endpoint
curl -X POST http://localhost:3000/api/webhooks \
  -H "Content-Type: application/json" \
  -d '{"url": "https://your-service.com/hooks/diagnosis"}'

# Diagnose with webhook delivery
curl -X POST http://localhost:3000/api/diagnose \
  -H "Content-Type: application/json" \
  -d '{"incident": "Redis cluster failover triggered in staging", "webhook": "https://your-service.com/hooks/diagnosis"}'

# Check webhook delivery history
curl http://localhost:3000/api/webhooks/https%3A%2F%2Fyour-service.com%2Fhooks%2Fdiagnosis/deliveries

Export Results & Audit Trail

Export any diagnosis as JSON or CSV for runbooks, postmortems, or compliance. The audit log tracks every action with timestamps and trace IDs.

# Export a diagnosis as JSON
curl http://localhost:3000/api/diagnose/DIAGNOSIS_ID/export?format=json

# Export as CSV
curl http://localhost:3000/api/diagnose/DIAGNOSIS_ID/export?format=csv

# Retrieve full audit log (last 100 events)
curl http://localhost:3000/api/audit-log?limit=100

Complete API Surface
